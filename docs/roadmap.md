# Live2dAssistant 迭代 Roadmap

## 1. 文档目的

这份 roadmap 用于把当前项目文档里的“方向描述”整理成可执行的阶段计划。

它服务于三件事：
- 给后续迭代提供统一优先级
- 避免同时推进过多分散功能
- 为需求拆分、开发排期和验收提供依据

本文以当前主文档为基础：
- 大窗口模式优先
- 资料库保持只读托管与检索定位
- Agent 要围绕“可用、可读、可完成”三层能力持续增强

## 2. 当前判断

项目已经完成了第一轮可用性搭建：
- 大窗口模式主交互链可用
- Agent 工具链主路径可用
- 本地资料、知识卡片、记忆系统都有基础链路

当前主矛盾不是“缺少更多功能入口”，而是以下能力仍不够完整：
- 回复呈现还是伪流式，主界面体感不够顺
- 本地资料尤其 PDF 还不能进入正文级检索
- Agent 工具调用能跑通，但离稳定完成多步任务还有距离
- 记忆和卡片已经存在，但还没有形成高质量知识闭环

因此，下一阶段不建议把精力放在分窗口页面、美化动画或继续扩展零散工具，而应优先补齐主体验闭环。

## 3. 迭代原则

### 3.1 主界面优先

所有直接影响主体验的能力，优先围绕大窗口模式建设，不围绕分窗口扩展。

### 3.2 先打通闭环，再增加功能

优先补齐已有链路中的断点和薄弱点：
- 输出展示
- 资料读取
- 检索质量
- Agent 执行稳定性

在这些问题没有改善前，不优先扩展新模块。

### 3.3 先增强已有数据源，再引入复杂基础设施

在资料库和记忆系统已经具备 JSON/索引基础的前提下，优先增强：
- 正文抽取
- 全文索引
- 语义召回
- 结果重排

不建议过早抽象成过重的通用知识平台。

### 3.4 最小测试前置

虽然自动化测试不必一开始全面铺开，但在流式输出、索引和 Agent 编排进入重构前，至少应建立最小回归测试，防止主链路反复回退。

## 4. 阶段路线

## Phase 1: 对话体验闭环

### 目标

把“大窗口模式”从“能用”推进到“顺畅可持续使用”。

### 建设重点

- 真流式输出链路
- 长回复与短回复的统一渲染策略
- Agent 结构化轨迹面板
- 失败、取消、重试、超时等状态呈现
- 主界面交互细节收口

### 建议交付

- 主模型响应支持真实流式回传
- 渲染层按 token 或分段实时更新消息
- Agent 面板至少展示：
  - 当前步骤
  - 已调用工具
  - 工具结果摘要
  - 错误状态
- 长回复展开逻辑与轨迹展示逻辑解耦

### 验收标准

- 用户发起提问后，界面能在短时间内出现连续输出，而非等待整段完成
- 用户能看懂 Agent 当前是否在搜索、读资料、提炼记忆或生成答案
- 工具调用失败时，界面不会只剩一段模糊 trace 文本

### 非目标

- 不在此阶段追求复杂视觉动画
- 不优先做分窗口正式功能页

## Phase 2: 资料可读到资料可用

### 目标

让资料库从“只支持预览”升级为“Agent 能真正读到核心内容”。

### 建设重点

- PDF 正文抽取
- PDF 纳入统一索引流程
- 文本、Markdown、代码、PDF 的统一读取接口
- 检索结果从文件级升级到片段级

### 建议交付

- 引入 PDF 文本抽取能力
- 资料索引记录正文片段而非仅摘要
- `read_library_file` 对 PDF 返回正文片段或抽取结果摘要
- 搜索结果支持定位到 chunk，而不是只返回文件

### 验收标准

- PDF 不仅能预览，也能被检索和引用
- Agent 回答资料相关问题时，能够引用正文片段而不只是文件名或摘要
- 索引增量更新不因 PDF 接入而明显拖垮启动体验

### 非目标

- 不在此阶段承担资料编辑能力
- 不做复杂文档协作能力

## Phase 3: 本地语义 RAG

### 目标

把当前“关键词召回”升级为“语义召回 + 重排”的本地知识检索体系。

### 建设重点

- 资料库 embedding
- 查询 embedding
- chunk 级语义召回
- 关键词召回与语义召回混合
- rerank 或规则重排

### 建议交付

- 保留现有增量索引结构，同时扩展 embedding 字段或并行向量索引
- 搜索流程改为：
  1. 关键词粗召回
  2. 语义召回补充
  3. 综合重排
- 为 Agent 提供更清晰的检索结果结构：
  - 来源文件
  - 命中片段
  - 命中原因
  - 相关度

### 验收标准

- 用户使用近义表达、摘要表达、问题表达时，依然能命中相关本地资料
- 检索结果前几位的相关性明显优于纯关键词搜索
- 语义检索不破坏现有增量更新机制

### 非目标

- 不追求一次性构建通用多租户知识平台
- 不优先做联网搜索与本地知识的统一大编排

## Phase 4: Agent 从可调用到可完成

### 目标

让 Agent 不只是“会调工具”，而是能更稳定地完成一段多步任务。

### 建设重点

- 任务拆解
- 步骤执行状态机
- 工具调用失败恢复
- 搜索结果深读
- 执行过程结构化记录

### 建议交付

- 将当前规则驱动工具调用，逐步升级成显式步骤执行
- 每一步至少具备：
  - 目标
  - 动作
  - 输入
  - 输出
  - 状态
- 联网搜索从“返回标题和链接”扩展到“可抓取并阅读结果页正文”
- 对常见失败场景增加降级策略

### 验收标准

- 面对“先搜索、再读资料、再总结”的任务，Agent 能稳定完成主要路径
- 工具失败后，Agent 能给出明确失败点，而不是整体失效
- 用户能够通过面板看清任务进行到哪一步

### 非目标

- 不追求复杂自治 Agent
- 不在此阶段引入难以验证的大规模自动规划系统

## Phase 5: 记忆与知识闭环

### 目标

把上下文、长期记忆、知识卡片、资料库连接成更稳定的知识沉淀系统。

### 建设重点

- 记忆去重
- 污染控制
- 记忆分类质量提升
- 卡片未保存保护
- 卡片和资料建立引用关系

### 建议交付

- 记忆增加去重、合并、失效和置信度回收机制
- 卡片编辑增加未保存离开保护
- 卡片记录来源：
  - 来自哪段对话
  - 来自哪份资料
  - 来自哪条记忆
- 资料搜索结果可一键沉淀为卡片草稿

### 验收标准

- 长期使用后，记忆列表不会快速失控膨胀
- 卡片能承担“沉淀结论”的作用，而不只是另一个文本框
- 用户能看清知识内容的来源和可信度

### 非目标

- 不优先做高度复杂的知识图谱
- 不优先做大规模可视化分析

## 5. 优先级建议

建议按以下顺序推进：

1. Phase 1 对话体验闭环
2. Phase 2 资料可读到资料可用
3. Phase 3 本地语义 RAG
4. Phase 4 Agent 从可调用到可完成
5. Phase 5 记忆与知识闭环

排序原因：
- Phase 1 直接影响主界面的第一印象和可用性
- Phase 2 决定 Agent 是否真的“读得到东西”
- Phase 3 决定本地资料检索质量是否足够支撑日常使用
- Phase 4 决定 Agent 是否能稳定完成复杂任务
- Phase 5 决定系统能否沉淀长期价值

## 6. 建议拆分方式

每个阶段都建议拆成三个层次推进：

### 6.1 基础能力

先补后端链路、数据结构、索引或状态流。

### 6.2 渲染展示

再补主界面展示、状态反馈和交互闭环。

### 6.3 数据验证

最后补最小回归测试、日志和异常场景验证。

## 7. 当前不建议优先投入的方向

- 分窗口正式功能页
- 更细的视觉动画
- 零散的新工具入口
- 纯外观型改版

原因不是这些没有价值，而是它们目前不解决主链路问题。

## 8. 最小工程保障建议

在 roadmap 执行前或 Phase 1 初期，建议补以下工程保障：

- 至少为索引、记忆、卡片、Agent 主链路建立最小测试
- 为关键 IPC 输入输出增加结构校验
- 为流式输出、索引构建、工具调用增加可追踪日志
- 为文档新增专项设计说明：
  - `docs/agent-design.md`
  - `docs/data-models.md`
  - `docs/renderer-structure.md`

## 9. 建议的近期执行顺序

如果只看最近一轮开发，建议按下面顺序落地：

1. 真流式输出和结构化轨迹面板
2. PDF 正文抽取和索引接入
3. 资料库 chunk 级检索改造
4. 本地语义召回与重排
5. Agent 多步执行状态机
6. 记忆与卡片的数据质量增强

## 10. 维护方式

当以下内容发生变化时，应同步更新本 roadmap：

- 主体验从大窗口模式转向其他入口
- 资料索引从关键词检索升级为其他主方案
- Agent 工具边界发生明显变化
- 记忆系统和卡片系统的数据结构发生变化

如果路线图进入执行阶段，建议在每个 phase 下继续增加：
- 负责人
- 任务拆分
- 里程碑日期
- 风险记录

## 11. 2026-03-04 执行进度更新

本节用于同步当前代码边界，避免 roadmap 继续把已完成项写成待实现项。

### 已完成

- Phase 1 已部分落地：
  - 真实流式输出主链路已接入
  - assistant 已支持取消、超时、重试
  - Agent 执行摘要与折叠式详情面板已接入
  - 普通聊天与 Agent 链路的人设来源已统一
- Phase 2 已部分落地：
  - PDF 正文抽取已接入
  - PDF 已进入 chunk 与索引链路
  - organizer 已能同时展示 PDF 抽取文本、命中片段和原文件
- 工程辅助能力已补充：
  - 设置页已有显式 Agent 自检入口

### 未完成

- Phase 1 未完成项：
  - 执行面板还不是完整任务树
  - 没有步骤耗时、参数级展示和工具级独立超时
  - 缺少系统化回归测试
- Phase 2 未完成项：
  - 扫描件 PDF 仍无 OCR
  - 检索仍是关键词 / 规则召回，尚未进入语义层
  - 检索结果排序和摘要质量还可以继续提升

### 顺序调整

基于当前落地状态，近期执行顺序调整为：

1. Phase 3 本地语义 RAG 第一版
2. Phase 4 Agent 多步执行状态机
3. Phase 1 执行面板增强（耗时、参数、失败分类）
4. Phase 2 检索排序与摘要质量优化
5. 最小回归测试补齐

### Agent 数据流补充

- Phase 1 在 Agent 主链路上新增了一项已落地能力：prompt 职责拆分。
- 当前已经不是“同一 prompt 同时负责工具编排和最终回答”，而是：
  - `agent planning prompt` 负责内部思考、工具调用和内部草稿
  - `assistant final answer prompt` 负责最终用户可见回答
- 当前 workflow 面板已保留摘要化思考过程，聊天气泡只保留最终助手回答。
- 这项调整的意义是：
  - 避免 tool JSON、测试式旁白和内部编排文本污染最终回复
  - 让 Agent 和 persona 的职责边界稳定下来

### 近期优先级修正

基于当前代码状态，近期优先级建议调整为：

1. Phase 1 执行面板与 workflow 文案继续收口
2. Phase 4 Agent 多步执行状态机与失败恢复
3. Phase 2 检索排序与摘要质量优化
4. Phase 3 本地语义 RAG 第一版
5. 最小回归测试补齐

调整原因：Agent 主链路当前已经进入“数据流收口阶段”，优先把 workflow 表达、失败恢复和资料检索质量做稳，比立刻接入语义 RAG 更符合当前产品边界。
